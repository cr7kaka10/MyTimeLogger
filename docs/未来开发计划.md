## 阶段 1（当前优先）：PC 端 — 柳比歇夫分类计时界面

> **策略**: 先做纯本地版本，不依赖服务端。后续阶段 2 部署服务端后再接入同步。

### 功能描述

仿 aTimeLogger Pro 的圆形图标网格界面，执行柳比歇夫时间管理法：

- **一级分类**: 输入（学习/阅读/编程...）、输出（写作/教学/创作...）、生活（运动/饮食/社交...）
- 每个分类有**自定义图标 + 颜色**
- 点击分类图标 → 调用已有的倒计时专注核心 (`logic.py` 的 `start_with_context()`)
- 计时结束后自动归类记录
- 本地 SQLite 存储分类和记录

## 阶段 2：服务端 API + Docker 部署

> **前置条件**: 阶段 1 完成，PC 端柳比歇夫功能本地验证通过。

### 部署目标硬件（已确认）

| 参数 | 实况 |
|------|------|
| 设备 | **FastRhino R66S** |
| 架构 | **ARMv8 (aarch64)** 四核 1416MHz |
| RAM | **2GB** (当前可用 1.8GB / 90%) |
| 平均负载 | 0.03 ~ 0.12（极低） |
| 固件 | OpenWrt R22.10.10 |
| 内核 | 4.19.245 |

> **结论**: 2GB RAM + 四核 ARM，跑 FastAPI + MySQL 绰绰有余。关键是做好资源隔离。

### 资源隔离策略（不影响上网）

```yaml
# docker-compose.yml 关键配置
services:
  api:
    deploy:
      resources:
        limits:
          memory: 256M        # 硬限制 256MB
          cpus: '0.5'         # 最多用 0.5 个核心
    restart: unless-stopped

  db:
    image: mysql:8.0
    deploy:
      resources:
        limits:
          memory: 384M        # MySQL 硬限制 384MB
          cpus: '0.5'
    command: >
      --innodb-buffer-pool-size=64M
      --max-connections=20
      --performance-schema=OFF
    volumes:
      - /mnt/sda1/mysql_data:/var/lib/mysql   # 外接存储
```

**总内存预算**: 256MB (API) + 384MB (MySQL) = **640MB**，仅占总 RAM 的 32%。  
**CPU 限制**: 两个容器各 0.5 核，合共 1 核，路由器剩余 3 核处理网络。

### 额外保障措施

1. **Docker 网络隔离**: 容器使用 `bridge` 网络，不干扰路由器 `br-lan`
2. **端口映射**: 只暴露 `8080:8000`（API），不占用常规端口
3. **OOM 保护**: Docker 有 OOM Score 调整，路由核心进程优先级更高
4. **外接存储**: MySQL 数据目录挂载到 USB/TF 卡，不写路由内置 Flash
5. **限速可选**: 可给 Docker bridge 网口做 QoS 限速

### 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| Web 框架 | FastAPI (Python) | 异步原生、WebSocket 内建 |
| 数据库 | MySQL 8.0 (Docker) | 已有 MySQL 配置可复用 |
| ORM | SQLAlchemy 2.0 + alembic | 异步 + Migration |
| 实时通信 | WebSocket (FastAPI 原生) | 多端状态推送 |
| 认证 | JWT (python-jose) | 轻量统一 |

## 阶段 6：多端实时同步 + 数据一致性

### 同步流程

```
PC 端 ──start──▶ WebSocket Hub ──推送──▶ 安卓端 (UI 更新)
安卓端 ──stop──▶ WebSocket Hub ──推送──▶ PC 端 (缓存刷新)
```

### 离线策略

- 断网时本地 Room/SQLite 暂存事件队列
- 联网后自动推送，冲突解决: Last-Write-Wins + 事件版本号
- 启动时先全量同步一次

## 风险与注意事项

| 风险 | 缓解措施 |
|------|---------|
| R66S 性能不够 | Docker 资源限制已设 (640MB / 1核)，留足余量给路由 |
| Docker 影响上网 | bridge 网络隔离 + CPU/MEM 硬限制 + 不占用常规端口 |
| ARM 镜像构建慢 | PC 上 `docker buildx` 交叉编译，`docker save/load` 传到路由 |
| 安卓后台计时被杀 | Foreground Service + 通知栏 + 引导用户加电池白名单 |
| WebSocket 断线 | 指数退避重连 + 离线事件队列 + 启动全量同步 |
| 外网访问需求 | DDNS + 端口转发 或 frp/Tailscale 内网穿透 |
