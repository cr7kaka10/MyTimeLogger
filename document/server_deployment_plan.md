# 睡眠分析服务端部署与数据同步方案 (v1.0)

## 1. 目标背景
当前睡眠分析（OCR + AI 复盘）依赖 PC 端软件开启。为了实现“全时在线”上传与分析，需将核心解析逻辑迁移至云端服务器，并解决云端数据与本地 PC 数据库的同步一致性问题。

## 2. 架构设计

### 2.1 云端服务 (Cloud Server)
- **技术栈**: Python + FastAPI / Flask
- **职责**:
    - 接收手机端上传的截图。
    - 调用 AI 视觉模型提取指标。
    - 调用 AI 文本模型生成深度报告。
    - 将结果存入 **云端 Buffer 库**（轻量级 SQLite 或 Redis）。
- **接口**:
    - `POST /upload`: 接收图片并触发异步分析。
    - `GET /sync_data`: 供 PC 端拉取未同步的数据。

### 2.2 数据同步策略：离线补偿拉取 (Polling & Pull)
由于 PC 端通常位于内网且不保证 24 小时在线，采用 **“客户端主动拉取”** 模式：

1. **增量记录**: 云端每条记录包含 `updated_at` 时间戳和全局唯一 ID（建议使用 `date`）。
2. **心跳同步**:
    - PC 端软件启动时，自动查询本地库中最新的 `updated_at` 或最近 7 天的记录状态。
    - 发起 `GET /sync_data?since={timestamp}` 请求。
3. **冲突处理**:
    - **覆盖原则**: 若本地已有该日期记录且 `updated_at` 早于云端，则更新本地。
    - **合并原则**: 若本地有 Reflection（评价）而云端没有，则保留本地评价，仅同步指标。
4. **清理机制**: 云端记录在同步成功 N 次或超过 30 天后自动清理，保持云端轻量。

## 3. 安全与稳定性

### 3.1 安全防护
- **Token 校验**: 接口需带上 `X-Auth-Token`，防止被恶意刷 API 导致扣费。
- **HTTPS 传输**: 全程加密，保护睡眠隐私数据。

### 3.2 稳定性保障
- **异步处理**: 上传后立即返回 `request_id`，后台排队分析，手机端通过 SSE 或 轮询获取状态。
- **重试机制**: 服务端内置 3-5 次 AI 调用重试逻辑。

## 4. 实施计划 (Roadmap)
1. **模块剥离**: 将 `AIWorker` 逻辑从 PyQt 线程中剥离，封装为纯 Python 的 `SleepAnalyzer` 类。
2. **接口开发**: 编写基于 FastAPI 的服务端包装器。
3. **PC 客户端补丁**:
    - 在 `StudyLogger` 中增加 `cloud_sync()` 方法。
    - 界面增加“☁️ 云端同步中”状态显示。
4. **MySQL 方案 (进阶)**: 若用户已有公网 MySQL，可直接将 `database.py` 的 `db_type` 切换为 `mysql`，实现天然的多端数据共享（无需额外编写同步逻辑）。
